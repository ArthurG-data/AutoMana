## API Layer

Source: `docs/api/API_LAYER_BACKLOG.md` (full senior-level review 2026-05-22).

### High Priority

**H1 — `schemas/auth/token.py` — `check_expiry` always sets `is_expired = False`**

`now < now + delta` is trivially true. Any code reading `token.is_expired` always sees `False` regardless of actual expiry.

- Severity: **Critical**
- File: `src/automana/api/schemas/auth/token.py`
- Fix: `if datetime.fromtimestamp(values.exp, tz=timezone.utc) > datetime.now(timezone.utc):`

---

**H2 — `auth_service.py:105` — service raises `HTTPException`**

`login` raises `HTTPException(401)` inside the service layer. Services must not import or raise HTTP-layer exceptions — breaks Celery/internal callers and makes the service untestable without an HTTP context.

- Severity: **High**
- File: `src/automana/api/services/auth/auth_service.py:105`
- Fix: Return `None` on bad credentials; router's existing `if result is None` guard already handles it.

---

**H3 — `password_reset_service.py:65–69` — service raises `HTTPException`**

`reset_password` raises `HTTPException(400)` for invalid/expired/used tokens inside the service layer.

- Severity: **High**
- File: `src/automana/api/services/auth/password_reset_service.py:65–69`
- Fix: Raise a domain exception (e.g. `InvalidResetTokenError`) and translate to 400 in the router.

---

**H4 — Four independent password-hashing implementations**

| Location | Library |
|----------|---------|
| `services/auth/auth.py` | raw `bcrypt` |
| `utils/auth.py` | passlib `CryptContext(bcrypt)` |
| `core/utils/get_hash_password.py` | passlib `CryptContext(bcrypt)` |
| `worker/user_creation.py` | imports from `core.utils.auth` |

Raw bcrypt and passlib bcrypt produce slightly different hash formats. A hash created by one path may silently fail verification by another.

- Severity: **High**
- Fix: Designate `core/utils/get_hash_password.py` as canonical; remove duplicates.

### eBay API Quotas & Limits

**P6 — Apply for eBay Partner Network (EPN) to raise Finding API quota from 5k to 50k–100k/day**

The Finding API default quota is **5,000 calls/day per App ID** (separate from the 9M platform ceiling shown in the developer portal, which is a shared total). At 500 cards × 3 marketplaces = 1,500 calls/run, the quota is fine today but becomes a hard wall at ~1,500 cards. Applying to EPN and requesting a call limit increase raises the per-App-ID quota to 50k–100k/day, removing the ceiling for foreseeable scale.

- Severity: **Medium** (not blocking now; blocking at ~1,500 watchlist cards)
- Fix: Apply at `partnernetwork.ebay.com`. After EPN acceptance, file a call limit increase at `developer.ebay.com → Contact Us → API call limit increase`, referencing EPN membership.
- Note: Multiple App IDs under the same developer account share the same quota pool — registering more keys does not help.
- Discovered: 2026-05-23

---

**P7 — Finding API: `totalEntries` ignored — silent data truncation beyond 100 results**

`find_completed_items` hardcodes `paginationInput.pageNumber: "1"` and never checks the `totalEntries` field in the response. For high-volume cards (e.g., playable commons) the real result set can exceed 100 items. Subsequent pages are silently dropped.

- Severity: **Low** (price estimates are still directionally correct; affects completeness, not correctness)
- File: `src/automana/core/repositories/app_integration/ebay/ApiFinding_repository.py`
- Fix: Read `totalEntries` from `result_block`; loop pages while `offset < totalEntries` and quota allows.
- Discovered: 2026-05-23

---

**P8 — Finding API: No scrape-frequency tiering — all cards scraped equally nightly**

All watchlist cards are scraped every night across all 3 marketplaces regardless of price velocity. A Reserved List staple sells 5–10 times/month globally; a playable uncommon sells 500+. Treating them identically wastes ~40–60% of the daily quota as the watchlist grows past 150 cards.

- Severity: **Low** (quota is not a problem at current scale)
- File: `src/automana/core/services/app_integration/ebay/scrape_global_market_service.py`
- Fix: Add `scrape_priority` (HIGH/MEDIUM/LOW) to `pricing.ebay_scrape_targets`. HIGH: nightly × 3 markets. MEDIUM: every 72h × US only. LOW: weekly × US only.
- Discovered: 2026-05-23

---

**P9 — Browse API: `search_items()` leaks httpx connection pool (missing `async with self`)**

`ApiBrowse_repository.search_items()` calls `self.send()` directly without `async with self:`. The base class fallback creates an `httpx.AsyncClient` on `self._client` and never closes it. Under Celery task load this accumulates open file descriptors and will eventually exhaust the connection pool.

- Severity: **High** (slow resource leak — will manifest under production load)
- File: `src/automana/core/repositories/app_integration/ebay/ApiBrowse_repository.py`
- Fix: Wrap the `send()` call in `async with self:` (same pattern as `ApiFinding_repository`).
- Discovered: 2026-05-23

---

**P10 — Browse API: No retry or backoff on 429 — quota exhaustion silently returns empty results**

`ApiBrowse_repository` has no rate-limit detection or retry. When the 5,000-call/day quota is exhausted, `market_price_service.py` catches the raised `EbayBuyApiRateLimitError` in a bare `except Exception` and silently returns empty `CardMarketData` to all users for the rest of the day — indistinguishable from a legitimate empty result set.

- Severity: **High** (silent data failure with no user-visible error and no circuit breaker)
- File: `src/automana/core/repositories/app_integration/ebay/ApiBrowse_repository.py`
- Fix: Add HTTP 429 detection with exponential backoff (2s base, 3 retries, ±0.5s jitter), mirroring the `_is_rate_limited` + retry loop already in `ApiFinding_repository.py`. Re-raise after max retries.
- Note: Browse API returns a proper HTTP 429 (unlike Finding API which embeds the error code in a 200/500 body).
- Discovered: 2026-05-23

---

**P11 — Browse API: `Content-Type: application/json` on GET requests; bad filter serialization**

Two code-quality issues in Browse API callers:
1. `market_price_service.py` and `browsing_services.py` send `Content-Type: application/json` headers on GET requests (which have no body). This violates HTTP semantics and is a latent risk with stricter API gateways.
2. `market_price_service.py` passes `filter` as a Python list `[f"conditionIds:{{{condition_id}}}"]` directly to httpx. Browse API expects a single comma-separated string. Works today with one-element lists but breaks silently with multiple filters.

- Severity: **Low**
- Files: `src/automana/core/services/app_integration/ebay/market_price_service.py`, `browsing_services.py`
- Fix: Remove `Content-Type` from GET headers; change `filter` to `",".join(filters)`.
- Discovered: 2026-05-23

## Infrastructure

### Caching

**I1 — Redis cache uses orjson-only serialization**

`RedisCache` serializes with orjson, which does not support custom Python objects. Callers must ensure all cached values are JSON-serializable; there is no graceful fallback.

- Severity: **Low**
- Source: `docs/infrastructure/CACHING.md` Known Limitations

---

**I2 — Pattern-matching cache invalidation is O(n) via `KEYS`**

`invalidate_pattern(pattern)` calls Redis `KEYS` — a blocking O(n) scan across all keys. This is safe at small scale but will degrade under high key volumes.

- Severity: **Low** (acceptable until key count grows)
- Source: `docs/infrastructure/CACHING.md` Known Limitations
- Fix: Replace with `SCAN` cursor approach for production at scale.

---

**I3 — No cache statistics or metrics**

Cache hit/miss rates are not tracked. There is no visibility into cache efficiency.

- Severity: **Low**
- Source: `docs/infrastructure/CACHING.md` Future Improvements
